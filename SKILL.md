# Easy Solution Architect

## 显式加载规则

- 本 skill 只能由用户显式点名加载：`$easy-solution-architect`、`easy-solution-architect`，或清楚表达“加载/使用 Easy Solution Architect skill”
- 不得因为用户提出技术方案、架构设计、Architect-Spec、实施级技术 Spec 或类似提示而自动加载
- 如果用户没有显式点名，应按普通 agent 流程处理，不应用本 skill 的追问、确认和 Architect-Spec 写入流程

加载后，当用户希望基于当前目录项目代码、`.easy-coding` 过程文档和本轮提示词，输出可直接指导开发的技术方案时，使用此 skill。

这个 skill 的核心工作方式是：

- 先基于证据分析，再决定是否追问
- 先静默消化已有信息，不提前展示大段分析内容
- 先给架构方案摘要，再等用户确认
- 用户确认后，才写入 `.easy-coding/spec/Architect-Spec.md`

如果信息不足以支撑关键技术决策，不要先写残稿；要先指出缺口，并给出带推荐项的问题。

## 对话硬规则

以下规则必须严格遵守：

- 在开始追问前，先静默分析项目代码、`.easy-coding` 文档和原型文件
- 对于已经能够确认、且没有冲突的信息，不要提前逐条展示给用户
- 在正式成文前，不要向用户输出大段“已分析数据展示”“长篇现状盘点”或接近完整 Spec 的内容
- 与用户对话时，只聚焦三类内容：
  - 冲突项
  - 缺失项
  - 需要用户拍板的高影响决策
- 追问必须使用 `A/B/C/D` 选项式表达，让用户可以直接回复字母
- 不要要求用户复制整段选项文字

正确节奏是：

1. 静默分析
2. 先生成问题规划
3. 按规划进行多轮追问与确认
4. 问题收敛后统一输出架构方案摘要
5. 用户确认后写入 `Architect-Spec.md`

## 显式加载后的适用场景

显式加载后，适用于以下诉求：

- 做技术方案、架构设计、技术 Spec、实施方案
- 输出或更新 `.easy-coding/spec/Architect-Spec.md`
- 结合 `Product-Spec.md`、`UI-Spec.md`、原型文档或项目代码做架构设计
- 分析当前项目的技术栈、目录分层、数据模型、核心模块或 DDL
- 基于现有项目与需求，补充 API、状态机、事件流、异步处理设计

这个 skill 不允许语义自动触发，只能由用户显式说出 `$easy-solution-architect`、`easy-solution-architect` 或明确要求加载 Easy Solution Architect skill。

## 默认产物

默认只生成或更新一个输出文件：

- `.easy-coding/spec/Architect-Spec.md`

固定约束：

- 默认语言为中文
- 默认文档粒度为实施级技术 Spec
- 若已有 `Architect-Spec.md`，优先增量更新并保留既有结构
- `Product-Spec.md`、`UI-Spec.md`、原型说明、HTML 原型、AI 图片原型与资源索引默认只读

## 输入发现

按以下顺序发现和使用输入：

1. 当前项目代码与目录结构
2. `.easy-coding/spec/Architect-Spec.md`
3. `.easy-coding/spec/Product-Spec.md`
4. `.easy-coding/spec/UI-Spec.md`
5. `.easy-coding/prototype/Easy-UI-Prototype.md`
6. `.easy-coding/prototype/index.html`、同目录页面 HTML、`assets/` 与 `images/` 中的原型产物索引
7. 用户当前提示词

读取规则：

- 先扫描全项目结构，再聚焦与用户需求相关的模块、包、应用或目录
- `.easy-coding` 缺失时，不要直接阻断；继续基于代码分析，并只对真正缺失的关键输入追问
- 原型输入只认 `.easy-coding/prototype/` 当前规范路径；若该路径或说明文档缺失，不要自动搜索旧路径或项目根目录下的原型文件，按“无原型输入”继续分析
- HTML 与图片原型只作为架构输入依据，不作为生产实现来源；图片模式没有可读 HTML 时，以 `.easy-coding/prototype/Easy-UI-Prototype.md` 中的页面说明、图片文件名和提示词摘要作为依据
- 能从代码、配置、目录、Spec、原型中推出的信息，不要反问用户
- 对可确认且无冲突的信息，默认内部消化，不主动做大段展示
- 需要查看固定文档骨架时，读取 [references/architect-spec-template.md](references/architect-spec-template.md)

## 冲突优先级

当多个来源冲突时，默认优先级如下：

1. 用户当前提示词
2. 当前代码现状
3. 历史 Architect / Product / UI / Prototype 文档

处理规则：

- 先用精简方式输出“冲突摘要”，只说明冲突点、当前倾向采用的版本、是否必须由用户拍板
- 对会影响技术路线、模型设计、接口形态、状态流转、异步机制的冲突，必须先确认再继续
- 不要悄悄覆盖历史设计结论

## 核心工作流

严格按以下顺序工作：

1. 发现输入源，并确认是否已有 `Architect-Spec.md`
2. 扫描项目技术栈、目录结构和相关模块
3. 读取 `.easy-coding` 中的 Product / UI / Prototype / Architect 文档
4. 从代码与文档中提炼：
   - 技术栈
   - 架构分层
   - 数据模型线索
   - 核心模块与设计模式线索
   - 已有约束、风险与冲突
5. 区分“可推断项”和“必须确认项”
6. 先生成“问题规划”，列出还需确认的问题主题、优先级和预计轮次
7. 按问题规划进行多轮追问，每轮不超过 3 个问题
8. 每轮回答后，更新问题规划，继续追问，直到关键疑虑全部解决
9. 当关键问题都收敛后，再统一输出一版结构化“架构方案摘要”
10. 用户确认后，再写入 `.easy-coding/spec/Architect-Spec.md`
11. 写完后，重新对照 `Product-Spec.md`、`UI-Spec.md`、原型说明、HTML 页面和图片页面索引做一轮冲突自检
12. 最后静默检查章节完整性、路径一致性和文档内自洽性

不要跳过第 6-9 步。用户未确认方案摘要前，不要写 `Architect-Spec.md`。
在第 9 步之前，不要输出接近完整 Spec 的中间稿。
只要仍有关键疑虑未解决，就不要输出摘要，更不要开始文档撰写。
在第 11 步发现明显冲突时，要及时反馈用户，不要把冲突静默吞掉。

## 可推断项与必须确认项

### 可推断项

以下内容优先从项目和文档中推断：

- 核心语言与版本
- 主框架、运行方式、构建方式
- 数据库和存储线索
- 前端视图框架、组件库、状态管理方式
- 目录分层和模块边界
- 已有模型、接口、事件、任务、状态对象

### 必须确认项

当以下任一项无法可靠推断时，必须追问：

- 关键技术栈无法确认
- 新需求会改变已有架构路线
- 数据持久化方案无法确认
- 模块边界涉及重大重构
- 原型表达与产品文档存在明显冲突
- API 形态、状态流转、异步机制会影响核心设计，但仓库中无法确认
- 用户提出了新的性能、安全、兼容性、审计等重要非功能约束

只要“必须确认项”里仍有未关闭项，就视为疑虑未解决。

## 问题规划规则

在正式提问前，先在内部生成一份问题规划，不要把完整内部推理直接展示给用户。

问题规划至少包含：

- 问题主题列表
- 每个问题主题为什么重要
- 哪些问题会阻断架构方案或文档输出
- 建议提问顺序

问题规划的使用规则：

- 先问最影响技术路线和文档正确性的主题
- 每轮最多问 3 个问题
- 一轮结束后，基于用户回答更新问题规划
- 如果某个问题已经被回答，不要换个说法重复追问
- 如果某个问题不再影响文档输出，可以从阻断项降级为备注项
- 当所有阻断项都关闭后，才允许进入摘要输出
- 不要为了“问得全面”而无限追问；当问题已经足够支撑实施级 Spec 时就停止

追问规则：

- 每次只问少量高价值问题
- 每轮最多问 3 个问题
- 每个问题都给 2-4 个推荐选项，并使用 `A/B/C/D` 标号
- 用户应当可以只回复字母完成选择
- 推荐项要结合项目规模、现状与用户目标
- 小项目优先推荐简单方案，例如纯 TypeScript 单体；中大项目再推荐更重的分层或中间件
- 不要要求用户复制整段选项内容
- 如果上一轮还有未回答问题，先等待用户回答，不要直接开启下一轮新问题

推荐提问格式：

```markdown
需要你拍板的 2 个点：

1. 数据持久化方案
A. 继续沿用现有 MySQL
B. 改为 SQLite 简化部署
C. 先以内存实现，后续再持久化
D. 其他（请补充）

推荐：A，原因是当前项目已有相关持久化痕迹，改动最小。

2. 模块划分方式
A. 保持单体分层
B. 按领域拆模块
C. 按前后端分仓
D. 其他（请补充）

推荐：B，原因是更适合后续扩展。
```

## 架构方案摘要门槛

进入写文档前，必须先向用户展示一版摘要。摘要至少包含：

- 需求理解
- 主要输入依据
- 已发现的冲突或风险
- 技术栈判断或建议
- 目录分层与模块划分
- 数据模型与持久化方案
- API 设计结论
- 状态机设计结论
- 事件流与异步设计结论

摘要输出前提：

- 所有阻断型问题都已关闭
- 不再存在会影响技术路线、数据模型、接口、状态机、异步机制的关键疑虑

如果关键问题尚未收敛，不要提前输出这份摘要。
不要一边留着待确认问题，一边说“这些不阻断我先写第一版”。

## Architect-Spec 写作规则

`Architect-Spec.md` 使用固定骨架。章节即使不适用也要保留标题，并明确填写“无”。固定骨架如下：

1. 需求背景与目标
2. 输入依据与冲突说明
3. 系统基础技术栈
4. 架构分层与目录职责
5. 数据模型设计
6. 核心模块设计
7. API 设计
8. 状态机设计（如需要）
9. 事件流与异步设计
10. 关键流程与交互链路
11. 非功能性设计
12. 实施建议与风险

详细模板与章节说明见 [references/architect-spec-template.md](references/architect-spec-template.md)。

写作要求：

- 系统基础技术栈必须明确核心语言、版本、主框架、数据库、前端视图框架、构建与运行方式
- 架构分层必须落到目录职责，而不是只写理念
- 数据模型必须区分表模型、持久化模型、领域模型、DTO/VO；传统数据库场景默认提供 DDL
- 核心模块设计要写职责边界、主要对象、调用关系和推荐模式
- API 设计、状态机设计、事件流与异步设计是常规模板章节；没有相关内容时必须写“无”

## 写入规则

在真正写入前，必须满足以下条件：

- 用户已确认架构方案摘要
- 关键冲突已处理或已明确记录
- 影响技术路线的关键未知项已确认
- 问题规划中的所有阻断项都已关闭

写入方式：

- 若 `.easy-coding/spec/Architect-Spec.md` 已存在，默认增量更新并尽量保留原结构
- 若文件不存在，则按固定骨架创建
- 不要顺手修改其他 Spec 或原型文件

## 验证规则

完成后静默检查：

- 文档是否包含固定骨架中的所有章节
- `API 设计`、`状态机设计（如需要）`、`事件流与异步设计` 是否存在
- 不适用章节是否明确写为“无”
- 路径、模块名、技术栈描述是否与代码和输入依据一致
- 若包含 DDL、接口或状态流转，是否与正文前后自洽
- 必须重新对照 `Product-Spec.md`、`UI-Spec.md`、原型说明、HTML 页面和图片页面索引，检查是否存在明显冲突

冲突复核规则：

- 只关注明显冲突，不做无限上纲的吹毛求疵式检查
- 重点检查：页面流程和模块设计是否冲突、核心对象和字段是否冲突、状态流转是否冲突、关键交互与 API / 异步设计是否冲突
- 图片模式没有可读 HTML 时，以 `.easy-coding/prototype/Easy-UI-Prototype.md` 中记录的页面说明、图片文件名和提示词摘要作为复核依据
- 如果没有明显冲突，可以继续正常交付，不必额外制造问题
- 如果发现明显冲突，必须在输出中单独反馈“冲突复核结果”，指出：
  - 冲突点
  - 冲突来源
  - 当前 `Architect-Spec` 采用的方案
  - 是否建议回头调整 Product / UI / Prototype 文档
- 冲突复核的目标是及时发现跨文档不一致，不是自动改写其他文档；默认仍然不直接修改 Product / UI / Prototype 文件

## 输出风格

- 优先引用事实，再给结论
- 不要泛泛而谈“建议采用最佳实践”
- 对推断出的高影响结论要说明依据
- 对无法确认的事项，要给推荐方案并明确请求用户确认

## 失败与降级处理

如果项目与 `.easy-coding` 信息都过少，导致无法支撑实施级 Spec：

- 明确说明缺了什么
- 给出推荐技术路线
- 提出最少量的关键问题
- 在用户确认前，不生成 `Architect-Spec.md`
